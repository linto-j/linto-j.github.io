100G Leaf-Spine Data Center Fabric (with Service Leaf / Border)

This project focused on architecting and deploying a production-grade 100G leaf-spine data center fabric using a Layer-3 underlay with BGP routing. The design was built to support high east-west throughput (GPU / compute workloads) while also providing a controlled and secure north-south path via a Service Leaf / Border for internet breakout and shared services.

100G Leaf-Spine L3 Underlay BGP Service Leaf / Border Segmentation High Availability

Executive Design Objectives

  • Deterministic routing and fast convergence (no STP dependency)
  • High east-west throughput for compute/GPU clusters
  • Controlled north-south traffic via a Service Leaf / Border layer
  • Multi-tenant segmentation with scalable VLAN/IP structure
  • High availability with no single point of failure
  • Operational simplicity: predictable troubleshooting and growth model

Fabric Roles

  • Spines: high-speed core; ECMP transit between leafs
  • GPU Leafs: top-of-rack leafs for compute/GPU nodes; optimized for east-west traffic
  • Service Leaf / Border: controlled north-south path to shared services and WAN/Internet breakout (often behind firewall/NAT)

Why L3 Fabric Instead of L2?

A Layer-3 fabric was selected to avoid Layer-2 scaling limitations (large broadcast domains, STP complexity) and to enable ECMP across multiple equal-cost routed paths. This provides deterministic forwarding behavior, simplifies troubleshooting, and allows predictable scale-out by adding more leaf switches.

Routing Model

The underlay uses BGP for route exchange between leaf and spine layers. Each leaf establishes BGP peering with all spines (or via a standardized fabric peering model), enabling full fabric reachability. ECMP distributes flows across equal-cost paths and maintains predictable convergence behavior during failures.

North-South vs East-West Traffic Flows

  • East-West (Compute ↔ Compute): stays within fabric; leaf-spine-leaf paths using ECMP
  • North-South (Compute ↔ Internet/WAN): steered toward the Service Leaf / Border, then to firewall/NAT and upstream
  • Shared Services: DNS/NTP/management or shared networks can be anchored near the service/border zone (policy controlled)

Segmentation Strategy

  • Workload segmentation aligned with tenant / environment needs
  • VLAN/IP structure designed for predictable expansion and isolation
  • Optional VRF-based separation for multi-tenant environments (where required)
  • Service/Bord​er layer used to enforce policy boundaries for north-south access

IP Addressing Strategy

  • Loopbacks per device for stable routing identifiers
  • Point-to-point links (often /31) for spine-leaf interconnects
  • Clear separation of underlay addressing vs tenant/service addressing
  • Consistent naming and allocation model for operational clarity

High Availability Model

  • Dual spines + redundant uplinks from each leaf
  • ECMP-based traffic distribution and resilience
  • Failure domain isolation at leaf level
  • Service Leaf / Border redundancy (where applicable) to avoid single choke point

Key Outcomes

  • High-throughput east-west performance suited to GPU-intensive workloads
  • Deterministic routing behavior with simplified operations
  • Controlled and secure north-south breakout via Service Leaf / Border
  • Scalable architecture designed for future expansion
  • Production-grade resiliency and predictable troubleshooting
